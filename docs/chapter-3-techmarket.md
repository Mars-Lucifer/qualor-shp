# Глава 3. Техническая реализация веб-приложения TechMarket

## 3.1. Общая информация и архитектура программного решения

### 3.1.1. Назначение и технологический профиль проекта

Проект `TechMarket` представляет собой веб-приложение класса e-commerce, ориентированное на публикацию, поиск, фильтрацию и заказ высокопроизводительных вычислительных решений. Предметная область включает каталог ноутбуков и мини-ПК, пользовательскую авторизацию, корзину, оформление заказа, административное управление товарами и работу с новостными публикациями. Архитектура проекта реализована на базе `Next.js` c использованием `App Router`, языка `TypeScript`, утилитарного CSS-фреймворка `Tailwind CSS`, ORM-слоя `Drizzle ORM` и встраиваемой реляционной СУБД `SQLite3`.

Выбор указанного стека обусловлен необходимостью создать компактную full-stack систему, в которой пользовательский интерфейс, прикладная логика и уровень доступа к данным располагаются в пределах единой кодовой базы. Для дипломного проекта подобная конфигурация обладает существенным преимуществом: она снижает инфраструктурную сложность, упрощает сопровождение и позволяет проследить непрерывный путь данных от пользовательского действия до записи в БД без введения дополнительного сетевого слоя между отдельными фронтенд- и бэкенд-репозиториями.

Ключевой особенностью проекта является использование `Next.js` не только как средства серверного рендеринга интерфейсов, но и как платформы для построения полного прикладного контура. В результате в одном приложении объединены:

- механизм маршрутизации страниц;
- реализация HTTP API через `Route Handlers`;
- серверные модули бизнес-логики;
- слой работы с БД;
- клиентские компоненты пользовательского интерфейса.

Такая модель особенно рациональна для интернет-магазина, так как каталог, карточка товара, авторизация, корзина и административный контур представляют собой тесно связанные подсистемы, использующие общий доменный контекст.

### 3.1.2. Обоснование выбора Next.js как full-stack платформы

Использование `Next.js` в проекте `TechMarket` следует рассматривать как архитектурное решение, позволяющее объединить фронтенд и бэкенд без потери структурированности. В традиционной схеме раздельного фронтенда и бэкенда необходимо поддерживать как минимум два независимых приложения: клиентское SPA и отдельный API-сервер. Подобный подход увеличивает объём конфигурации, дублирует модели данных и усложняет межслойное согласование типов.

В рассматриваемом проекте эти риски минимизированы. Маршруты страниц и API размещены в единой файловой структуре `src/app`, а типы запросов и ответов синхронизированы с доменными сущностями, определёнными в `src/server`. Это снижает вероятность рассинхронизации между интерфейсом и серверной логикой.

Данное решение обусловлено следующими факторами:

1. `Next.js` поддерживает гибридную модель рендеринга. Платформа позволяет использовать CSR, SSR, SSG и ISR в рамках одной системы.
2. В рамках интернет-магазина это особенно важно, поскольку публичные страницы каталога и карточек товара предъявляют требования к индексации, скорости первой отрисовки и корректной работе при прямом переходе по URL.
3. Единый роутинг устраняет необходимость отдельно проектировать адресное пространство фронтенда и API. Например, маршруты `/catalog`, `/item/[id]`, `/basket` и `/api/products`, `/api/cart`, `/api/orders` принадлежат одной и той же архитектурной модели.
4. Встроенные средства оптимизации `Next.js` позволяют в дальнейшем расширять приложение без смены фреймворка: возможно внедрение серверных компонентов, кэширования, сегментной загрузки, оптимизации изображений и инкрементальной регенерации статического контента.

Следует отметить, что текущая версия `TechMarket` активно использует клиентские компоненты и загрузку данных через HTTP-запросы к собственным `Route Handlers`. Однако сам выбор `Next.js` остаётся оправданным, поскольку:

- сохраняется возможность перевода части публичных страниц в SSR/ISR-режим без полной переработки приложения;
- единая кодовая база уже содержит все необходимые серверные модули;
- прикладной код готов к расширению в сторону более выраженной серверной оптимизации.

Таким образом, `Next.js` в проекте выступает не только UI-фреймворком, но и интеграционной платформой, объединяющей представление, серверные операции и слой данных.

### 3.1.3. Архитектура каталогов и организация модулей App Router

Структура проекта организована в соответствии с принципами `App Router`, при котором маршрутизация определяется файловой системой. Основные директории имеют следующий состав:

```text
src/
  app/
    api/
    admin/
    auth/
    basket/
    catalog/
    item/[id]/
    components/
    lib/
  server/
    db/
  styles/
```

Архитектурная единица `src/app` выполняет функцию слоя представления и HTTP-маршрутизации. Внутри неё размещаются:

- страницы пользовательского интерфейса;
- клиентские провайдеры состояния;
- переиспользуемые UI-компоненты;
- обработчики API.

Показательным примером является корневой layout:

```tsx
export default function RootLayout({ children }: { children: ReactNode }) {
  return (
    <html lang="ru">
      <body>
        <AuthProvider>
          <CartProvider>{children}</CartProvider>
        </AuthProvider>
      </body>
    </html>
  );
}
```

Данный фрагмент демонстрирует, что глобальные клиентские контексты авторизации и корзины подключаются один раз на уровне корневой компоновки. Благодаря этому любой дочерний маршрут получает доступ к данным пользователя и актуальному состоянию корзины без дублирования иерархии провайдеров.

Подкаталог `src/app/api` реализует внутренний HTTP API. Каждый файл `route.ts` соответствует отдельной конечной точке. Например:

- `src/app/api/products/route.ts` обслуживает выборку каталога;
- `src/app/api/products/[id]/route.ts` возвращает детальную информацию о товаре;
- `src/app/api/cart/items/route.ts` добавляет товар в корзину;
- `src/app/api/orders/checkout/route.ts` оформляет заказ;
- `src/app/api/auth/login/route.ts` и `register/route.ts` реализуют авторизацию и регистрацию.

Каталог `src/server` выполняет функцию доменно-сервисного слоя. Здесь размещены:

- `db` — инициализация БД и декларация схемы;
- `auth.ts` — авторизация, управление сессиями, извлечение пользователя;
- `catalog.ts` — логика каталога, фильтрации, популярных товаров и новостей;
- `orders.ts` — корзина, оформление заказа, отзывы, административная работа с заказами;
- `http.ts` — общие средства валидации и обработки ошибок;
- `security.ts` — криптографические операции.

Такое разбиение является важным с точки зрения архитектурной чистоты. Компоненты интерфейса не взаимодействуют с базой данных напрямую. Вместо этого они обращаются к `Route Handlers`, а те делегируют вычисления специализированным серверным модулям. Данный механизм реализации предполагает разделение ответственности между:

- UI-слоем;
- транспортным API-слоем;
- прикладным сервисным слоем;
- слоем хранения данных.

### 3.1.4. Роль TypeScript в обеспечении типобезопасности

Использование `TypeScript` является одним из ключевых факторов надёжности проекта. Типизация присутствует на всех уровнях: в компонентах, в API-клиенте, в серверной логике и в схеме БД. Это позволяет сократить количество ошибок, связанных с передачей неполных или некорректных данных.

В клиентском API-модуле определены типы основных сущностей:

```ts
export interface ProductListItem {
  id: number;
  name: string;
  price: number;
  category: ProductCategory;
  brandId: number;
  brandName: string;
  screenInches: number | null;
  processor: ProcessorType | null;
  ramGb: number | null;
  storageGb: number | null;
  graphicsType: GraphicsType;
  graphicsModel: string | null;
  image: string | null;
  rating: number;
  reviewCount: number;
  orderCount: number;
}
```

Подобные интерфейсы выполняют сразу несколько функций:

- формируют контракт между сервером и клиентом;
- упрощают разработку компонентов;
- позволяют средству статического анализа заранее выявлять несовместимости типов.

Типизация также глубоко интегрирована в слой БД. Например, перечислимые поля описаны через ограниченные множества значений:

```ts
category: text('category', { enum: PRODUCT_CATEGORIES }).notNull(),
processor: text('processor', { enum: PROCESSOR_TYPES }),
graphicsType: text('graphics_type', { enum: GRAPHICS_TYPES }).notNull(),
```

Такой подход устраняет значительную часть ошибок предметной области. Например, категория товара не может быть произвольной строкой: она ограничена константным набором значений, синхронизированным между фронтендом, сервером и базой данных.

## 3.2. Реализация серверной части и модели данных

### 3.2.1. Общая организация серверного слоя

Серверная часть `TechMarket` реализована внутри того же приложения `Next.js`, что и пользовательский интерфейс. Архитектурно это выражается в использовании `Route Handlers` как транспортного уровня и отдельных модулей `src/server` как доменно-сервисного ядра.

Такой подход имеет ряд преимуществ:

- бизнес-логика не смешивается с компонентами интерфейса;
- API можно расширять независимо от структуры страниц;
- работа с БД сосредоточена в одном месте;
- авторизация, проверка прав доступа и валидация запросов централизованы.

В каждом маршруте обработка строится по единой схеме:

1. принимается `NextRequest`;
2. извлекаются параметры пути или тело запроса;
3. производится авторизация пользователя при необходимости;
4. вызывается серверный сервисный метод;
5. результат возвращается через `NextResponse.json(...)`;
6. исключения преобразуются в стандартизированный JSON-ответ через `handleRouteError(...)`.

Характерный пример:

```ts
export async function POST(request: NextRequest) {
  try {
    const user = requireAuthenticatedUser(request);
    const body = await readJson<CheckoutBody>(request);

    return NextResponse.json({ order: checkout(user.id, body) }, { status: 201 });
  } catch (error) {
    return handleRouteError(error);
  }
}
```

Из данного фрагмента видно, что route handler не содержит прикладных деталей оформления заказа. Его задача ограничена транспортной координацией: проверить наличие авторизации, десериализовать JSON и передать управление модулю `orders.ts`.

### 3.2.2. Инициализация SQLite и инфраструктура доступа к данным

В проекте используется `SQLite3` через драйвер `better-sqlite3`. Данная СУБД выбрана как компактное, локальное и достаточно производительное решение для дипломного проекта и демонстрационного интернет-магазина. В отличие от тяжёлых клиент-серверных СУБД, `SQLite` не требует отдельного процесса, что упрощает развёртывание и локальную разработку.

Инициализация базы описана в `src/server/db/index.ts`. База располагается по пути `storage/shop.db`, а каталог `storage` создаётся автоматически при запуске приложения:

```ts
const DATABASE_DIRECTORY = path.join(process.cwd(), 'storage');
export const DATABASE_PATH = path.join(DATABASE_DIRECTORY, 'shop.db');

if (!fs.existsSync(DATABASE_DIRECTORY)) {
  fs.mkdirSync(DATABASE_DIRECTORY, { recursive: true });
}
```

После создания соединения задаются важные настройки `SQLite`:

```ts
sqlite.pragma('journal_mode = WAL');
sqlite.pragma('foreign_keys = ON');
```

Режим `WAL` повышает устойчивость к конкурентным операциям чтения и записи, а включение `foreign_keys` гарантирует соблюдение ссылочной целостности. Для интернет-магазина это критично, поскольку удаление пользователя, товара или заказа должно корректно отражаться в связанных таблицах.

Особое значение имеет то, что проект использует одновременно:

- декларативную схему `Drizzle ORM`;
- ручную инициализацию таблиц через SQL в `createTables()`.

Данное решение обусловлено желанием обеспечить запуск проекта без отдельной системы миграций. При старте приложения таблицы и индексы создаются автоматически, а начальные данные заполняются функциями `seedRootAdmin()` и `seedNews()`.

### 3.2.3. Анализ схемы базы данных Drizzle ORM

Схема БД описана в `src/server/db/schema.ts`. Она охватывает весь жизненный цикл предметной области: пользователей, авторизацию, каталог, корзину, заказы, отзывы и кеш популярных товаров.

#### Таблица `users`

Таблица `users` хранит информацию о зарегистрированных пользователях:

- `id` — первичный ключ;
- `login` — отображаемый логин;
- `loginNormalized` — нормализованная форма логина для поиска и проверки уникальности;
- `name` — имя пользователя;
- `passwordHash` — криптографический хеш пароля;
- `role` — роль (`user` или `admin`);
- `createdAt` — время создания.

Наличие отдельного поля `loginNormalized` является важным проектным решением. Механизм реализации предполагает нормализацию логина к нижнему регистру и устранение неоднозначностей при сравнении. За счёт этого исключается регистрация двух пользователей с логинами, различающимися только регистром символов.

#### Таблица `sessions`

Таблица `sessions` реализует серверное хранение сессий:

- `userId` — ссылка на пользователя;
- `tokenHash` — хеш токена сессии;
- `expiresAt` — срок действия;
- `createdAt` — дата создания.

Связь `sessions.userId -> users.id` настроена с `onDelete: 'cascade'`, что гарантирует удаление сессий при удалении пользователя. Уникальный индекс по `tokenHash` не допускает коллизий идентификаторов сессий.

#### Таблица `brands`

Таблица `brands` выделяет бренды в самостоятельную сущность. Это обеспечивает нормализацию предметной области и устраняет дублирование текстового названия бренда в каждой записи товара. Поле `nameNormalized` используется для устойчивого поиска и повторного использования уже существующего бренда.

#### Таблица `products`

Таблица `products` является центральной сущностью каталога. Она содержит:

- базовые атрибуты товара (`name`, `price`, `category`);
- ссылку на бренд (`brandId`);
- параметры конфигурации (`screenInches`, `processor`, `ramGb`, `storageGb`);
- графические параметры (`graphicsType`, `graphicsModel`);
- поля аудита (`createdAt`, `updatedAt`);
- поле `nameSearch`, представляющее собой нормализованную строку для поиска.

Важной особенностью является использование разных типов данных:

- `integer` — для идентификаторов, цены, объёма памяти и временных меток;
- `real` — для диагонали экрана;
- `text` — для строковых атрибутов и перечислимых доменных значений.

Наличие индексов по `category`, `brandId`, `processor`, `graphicsType` и `nameSearch` обусловлено характером пользовательских запросов. Фильтрация каталога строится именно на этих полях, поэтому индексация уменьшает стоимость выборок.

#### Таблица `product_images`

Таблица `product_images` реализует отношение «один ко многим» между товаром и набором изображений. Поля:

- `productId` — внешний ключ на товар;
- `url` — путь к изображению;
- `sortOrder` — порядок отображения;
- `createdAt` — дата добавления.

Архитектурная единица выполняет функцию хранения медиаданных отдельно от основной карточки товара. Это позволяет поддерживать галерею изображений без усложнения структуры таблицы `products`.

#### Таблица `reviews`

Таблица `reviews` содержит пользовательские оценки товара:

- `productId` — ссылка на товар;
- `userId` — ссылка на автора отзыва;
- `rating` — оценка от 1 до 5;
- `comment` — текстовый комментарий;
- `createdAt` — дата создания.

Особенно важен уникальный индекс:

```ts
uniqueIndex('reviews_user_product_unique').on(table.userId, table.productId)
```

Он запрещает повторную оценку одного и того же товара одним пользователем, что делает агрегированные рейтинги более достоверными.

#### Таблица `cart_items`

Таблица `cart_items` реализует серверную корзину:

- `userId` — владелец корзины;
- `productId` — товар;
- `quantity` — количество;
- `createdAt`, `updatedAt` — временные метки.

Уникальный индекс `(userId, productId)` гарантирует, что один товар не будет дублироваться несколькими строками в корзине одного пользователя. При повторном добавлении товара увеличивается поле `quantity`, а не создаётся новая запись.

#### Таблицы `orders` и `order_items`

Оформление заказа разделено на две таблицы:

- `orders` — общая информация о заказе;
- `order_items` — состав заказа.

Таблица `orders` содержит пользователя, статус, итоговую стоимость и контактные данные доставки:

- `status` принимает значения `pending` или `shipped`;
- `totalPrice` хранит суммарную стоимость;
- `fullName`, `phone`, `email`, `address` фиксируют реквизиты получения.

Таблица `order_items` использует принцип денормализованного снимка товара на момент покупки. Помимо `productId`, она хранит:

- `productName`;
- `productPrice`;
- `productCategory`;
- `brandName`;
- технические характеристики;
- `imageUrl`.

Такое дублирование является осознанным проектным решением. Если товар впоследствии будет изменён или удалён из каталога, заказ должен сохранять исторически корректную информацию о приобретённой позиции.

#### Таблица `popular_products_cache`

Таблица `popular_products_cache` служит для кеширования популярных товаров по нескольким витринам:

- рабочие ноутбуки;
- игровые ноутбуки;
- мини-ПК.

Она хранит категорию популярности, идентификатор товара, его позицию в подборке и время расчёта. Механизм кеширования снижает стоимость повторных вычислений при отображении главной страницы.

### 3.2.4. Связи между сущностями

Схема БД формирует несколько ключевых отношений:

- `users -> sessions` как «один ко многим»;
- `brands -> products` как «один ко многим»;
- `products -> product_images` как «один ко многим»;
- `users -> reviews` и `products -> reviews` как «один ко многим»;
- `users -> cart_items` и `products -> cart_items` как «один ко многим»;
- `users -> orders` как «один ко многим»;
- `orders -> order_items` как «один ко многим».

Эти связи дополнительно усилены ограничениями удаления:

- `cascade` применяется там, где зависимые данные теряют смысл без родительской сущности;
- `restrict` предотвращает удаление бренда, если на него ссылаются товары;
- `set null` используется в `order_items.productId`, чтобы сохранить историческую строку заказа даже при удалении исходного товара.

Подобная стратегия обеспечивает согласованность модели данных и корректно отражает бизнес-логику интернет-магазина.

### 3.2.5. Реализация API через Route Handlers

API проекта реализовано в формате `Route Handlers` внутри `src/app/api`. С практической точки зрения это означает, что обработчики HTTP-запросов встроены в ту же систему маршрутизации, что и страницы интерфейса.

Пример обработки списка товаров:

```ts
export async function GET(request: NextRequest) {
  try {
    const query = parseProductFilters(request.nextUrl.searchParams);
    const result = listProducts(query);

    return NextResponse.json(result);
  } catch (error) {
    return handleRouteError(error);
  }
}
```

Данный механизм реализации предполагает:

1. чтение query-параметров;
2. преобразование строковых значений в доменную модель фильтров;
3. выполнение серверного запроса к БД;
4. возврат JSON-ответа;
5. унифицированную обработку ошибок.

Аналогичный шаблон используется и в защищённых маршрутах. Например, работа с корзиной начинается с проверки аутентификации:

```ts
const user = requireAuthenticatedUser(request);
```

При отсутствии пользователя генерируется исключение `HttpError(401, ...)`, которое затем преобразуется в JSON-ответ соответствующего HTTP-статуса.

Таким образом, API-слой в `TechMarket` реализован в форме компактных, но хорошо структурированных конечных точек, каждая из которых делегирует основную предметную работу специализированному модулю.

### 3.2.6. Валидация входных данных и унификация ошибок

Значительная часть устойчивости серверной части достигается через централизованный модуль `src/server/http.ts`. Он содержит:

- класс `HttpError`;
- функцию `readJson<T>()` для безопасственного чтения JSON;
- функции `parsePositiveInteger`, `parsePositiveNumber`;
- функции `requireNonEmptyString`, `optionalTrimmedString`;
- обработчик ошибок `handleRouteError(...)`.

Пример:

```ts
export function parsePositiveInteger(value: unknown, fieldName: string) {
  const parsed =
    typeof value === 'number'
      ? value
      : typeof value === 'string'
        ? Number(value)
        : Number.NaN;

  if (!Number.isInteger(parsed) || parsed <= 0) {
    throw new HttpError(400, `${fieldName} должно быть положительным целым числом`);
  }

  return parsed;
}
```

Данное решение обусловлено необходимостью вынести повторяющуюся проверку данных из бизнес-логики. Благодаря этому функции каталога, корзины и заказа работают уже с нормализованными значениями.

### 3.2.7. Механизм безопасности и использование модуля crypto

Подсистема безопасности проекта сосредоточена в `src/server/security.ts` и `src/server/auth.ts`. Для защиты паролей и сессионных токенов используется стандартный модуль Node.js `crypto`.

#### Хеширование паролей

Для хранения пароля применяется схема:

1. генерируется случайная соль длиной 16 байт;
2. выполняется функция `scryptSync(password, salt, 64)`;
3. сохраняется строка формата `salt:hash`.

Код реализации:

```ts
export function hashPassword(password: string) {
  const salt = randomBytes(16).toString('hex');
  const passwordHash = scryptSync(password, salt, 64).toString('hex');

  return `${salt}:${passwordHash}`;
}
```

Применение `scrypt` является оправданным выбором для системы аутентификации, так как алгоритм специально разработан для устойчивости к перебору за счёт вычислительной и памяти-зависимой нагрузки. Это значительно безопаснее хранения пароля в открытом виде или использования простых хеш-функций без соли.

#### Проверка пароля

При авторизации выполняется повторное вычисление хеша и сравнение через `timingSafeEqual(...)`:

```ts
const incomingHash = scryptSync(password, salt, 64);
const storedHash = Buffer.from(savedPasswordHash, 'hex');
return timingSafeEqual(incomingHash, storedHash);
```

Использование `timingSafeEqual` защищает от атак, основанных на анализе времени сравнения строк.

#### Сессии и токены

После успешной авторизации создаётся случайный токен:

```ts
const token = randomBytes(32).toString('hex');
```

Однако в таблице `sessions` сохраняется не сам токен, а его SHA-256-хеш:

```ts
export function hashSessionToken(token: string) {
  return createHash('sha256').update(token).digest('hex');
}
```

Подобное решение повышает безопасность, поскольку компрометация базы данных не даёт злоумышленнику немедленного доступа к активным сессиям. Клиент получает исходный токен через cookie `tech_market_session`, а сервер при каждом запросе вычисляет его хеш и сравнивает с данными таблицы `sessions`.

#### Cookie-политика

Cookie сессии устанавливается с параметрами:

- `httpOnly: true` — недоступность из JavaScript;
- `sameSite: 'lax'` — базовая защита от CSRF;
- `secure` в production-среде;
- `expires` на дату истечения сессии.

Такой набор атрибутов соответствует базовым требованиям к безопасной cookie-аутентификации.

Клиентский слой построен на компонентной модели React. Наиболее характерными примерами переиспользуемых UI-компонентов являются `Button`, `Input` и `ProductCard`.

- интерактивную работу с корзиной.

Особенностью реализации является встроенная логика интеграции с контекстом корзины:

const { getProductQuantity, addProduct, incrementProduct, decrementProduct } = useCart();

Таким образом, компонент не только отображает данные, но и адаптирует своё состояние к текущему содержимому корзины. Если товар уже добавлен, кнопка покупки преобразуется в индикатор количества.

#### Компонент `CartQuantityControl`

Компонент `CartQuantityControl` отвечает за изменение количества товара. При `quantity <= 0` он показывает кнопку добавления, а при положительном количестве — компактный stepper с кнопками `+` и `-`. Это позволяет использовать одну и ту же UI-единицу и в карточке товара, и на странице корзины.

### 3.3.3. Динамические страницы и маршруты с параметром `[id]`

В проекте используется механизм динамических сегментов `App Router`. Наиболее важный пример — маршрут `src/app/item/[id]/page.tsx`, представляющий карточку товара.

На клиенте идентификатор извлекается через `useParams()`:

```tsx
const params = useParams<{ id: string }>();
apiRequest<{ item: ProductDetail }>(`/api/products/${params.id}`)
```

На стороне API параметр обрабатывается в `src/app/api/products/[id]/route.ts`:

```ts
const { id } = await context.params;
const product = getProductById(parsePositiveInteger(id, 'ID товара'));
```

Механизм реализации предполагает двухступенчатую обработку:

1. сегмент URL передаёт идентификатор выбранного товара;
2. серверный маршрут валидирует идентификатор и извлекает детальную запись из БД.

Сама страница товара строится как клиентский компонент и загружает:

- основную информацию о товаре;
- галерею изображений;
- агрегированный рейтинг;
- количество отзывов;
- текущее количество товара в корзине пользователя.

Дополнительно реализовано переключение изображений по кругу, что делает страницу функционально полноценной товарной карточкой.

### 3.3.4. Механика работы корзины

Корзина реализована как совокупность:

- серверной таблицы `cart_items`;
- контекста `CartProvider`;
- API-маршрутов `/api/cart`, `/api/cart/items`, `/api/cart/items/[productId]`;
- UI-компонентов `ProductCard` и `CartQuantityControl`.

В `CartProvider` поддерживаются методы:

- `refreshCart()`;
- `addProduct(productId)`;
- `incrementProduct(productId)`;
- `decrementProduct(productId)`;
- `removeProduct(productId)`.

Общий механизм изменения корзины инкапсулирован в функции `mutateCart(...)`:

```tsx
const url = method === 'POST' ? '/api/cart/items' : `/api/cart/items/${productId}`;
const response = await apiRequest<CartResponse>(url, {
  method,
  body: method === 'DELETE' && !body ? undefined : JSON.stringify(body ?? { productId }),
});
setCart(response);
```

Это означает, что после каждой операции клиент получает от сервера полное актуальное состояние корзины. Такой подход исключает расхождение между локальным и серверным представлением данных.

Серверная логика корзины, в свою очередь, реализует три ключевые операции:

- добавление товара;
- удаление товара;
- изменение количества.

При повторном добавлении товара запись не дублируется, а количество увеличивается:

```ts
if (existingCartItem) {
  db.update(cartItems).set({
    quantity: existingCartItem.quantity + 1,
    updatedAt: Date.now(),
  })
} else {
  db.insert(cartItems).values({ ... })
}
```

Подобная реализация соответствует стандартной логике интернет-магазина и опирается на уникальный индекс `(userId, productId)`.

## 3.4. Реализация бизнес-логики приложения

### 3.4.1. Управление состоянием на клиенте

В проекте `TechMarket` управление состоянием построено на `React Context`, а не на `Zustand` или `LocalStorage`. Это подтверждается наличием двух ключевых провайдеров:

- `AuthProvider`;
- `CartProvider`.

`AuthProvider` хранит:

- текущего пользователя `user`;
- флаг готовности `ready`;
- методы `refreshUser`, `setUser`, `logout`.

`CartProvider` хранит:

- текущее состояние корзины `cart`;
- индикаторы `ready` и `isLoading`;
- вычисляемое значение `itemCount`;
- методы синхронизации корзины с сервером.

Выбор `Context API` оправдан с точки зрения масштаба проекта. Объём глобального состояния ограничен двумя доменами: пользовательской сессией и корзиной. Подключение отдельной внешней библиотеки для менеджмента состояния в таком случае не является необходимым.

Отсутствие `LocalStorage` также является осознанным решением. Корзина хранится на сервере и привязана к авторизованному пользователю, а не к конкретному браузеру. Это обеспечивает:

- консистентность данных между сессиями;
- возможность получать актуальную корзину после повторного входа;
- упрощение логики оформления заказа.

### 3.4.2. Поток данных: от действия пользователя до записи в SQLite

Одним из ключевых аспектов системы является целостный путь данных от интерфейса к базе. Наиболее показателен сценарий оформления заказа.

#### Этап 1. Действие на клиенте

Пользователь на странице корзины вводит данные получателя и отправляет форму:

```tsx
const response = await apiRequest<{ order: { id: number } }>('/api/orders/checkout', {
  method: 'POST',
  body: JSON.stringify(form),
});
```

#### Этап 2. Маршрут API

Запрос поступает в `src/app/api/orders/checkout/route.ts`, где:

- проверяется авторизация;
- JSON преобразуется в типизированный объект;
- вызывается функция `checkout(user.id, body)`.

#### Этап 3. Серверная бизнес-логика

В `src/server/orders.ts` функция `checkout(...)`:

1. валидирует поля `fullName`, `phone`, `email`, `address`;
2. извлекает актуальные товары из корзины пользователя;
3. вычисляет `totalPrice`;
4. в транзакции создаёт запись в `orders`;
5. формирует связанные записи `order_items`;
6. очищает корзину пользователя;
7. инвалидирует кеш популярных товаров.

Ключевой фрагмент:

```ts
const orderId = db.transaction((tx) => {
  const cartRows = tx.select(...).from(cartItems)...

  const orderInsertResult = tx.insert(orders).values({
    userId,
    status: 'pending',
    totalPrice,
    fullName: payment.fullName,
    phone: payment.phone,
    email: payment.email,
    address: payment.address,
  }).run();

  tx.insert(orderItems).values(...)
  tx.delete(cartItems).where(eq(cartItems.userId, userId)).run();

  return Number(orderInsertResult.lastInsertRowid);
});
```

Использование транзакции является принципиально важным. Если одна из операций завершится ошибкой, система не попадёт в промежуточное неконсистентное состояние, при котором заказ создан, а корзина не очищена, либо наоборот.

#### Этап 4. Запись в SQLite

После успешного завершения транзакции данные физически сохраняются в `SQLite`. Затем route handler возвращает клиенту JSON с идентификатором заказа, а интерфейс перенаправляет пользователя на страницу успешного оформления.

Таким образом, путь данных реализован как последовательная цепочка:

`UI-форма -> apiRequest -> Route Handler -> серверный сервис -> Drizzle ORM -> SQLite -> JSON-ответ -> обновление интерфейса`.

### 3.4.3. Реализация фильтрации каталога

Подсистема фильтрации — одна из наиболее развитых частей бизнес-логики проекта. На клиенте страница каталога формирует набор query-параметров, исходя из текущего состояния фильтров:

```tsx
if (filters.priceFrom) params.set("priceFrom", filters.priceFrom);
if (filters.priceTo) params.set("priceTo", filters.priceTo);
if (filters.processor) {
  const processor = labelToProcessor(filters.processor);
  if (processor) params.set("processor", processor);
}
for (const brand of filters.brands) {
  params.append("brand", brand);
}
```

На сервере эти параметры обрабатываются функцией `parseProductFilters(...)`, а затем преобразуются в SQL-условия в `buildWhereConditions(...)`.

Фильтрация поддерживает:

- категорию;
- сортировку;
- диапазон цены;
- диапазон диагонали экрана;
- бренд;
- процессор;
- объём оперативной памяти;
- объём накопителя;
- тип графики;
- текстовый поиск по названию.

Особое внимание заслуживает логика условных фильтров. Например, для категории `mini_pc` отключаются параметры диагонали экрана как нерелевантные:

```ts
function categorySupportsScreen(category: ProductCategory) {
  return category === 'laptop';
}
```

Данное решение обусловлено стремлением сделать фильтрацию предметно-корректной: пользователю не предлагаются параметры, не относящиеся к выбранному типу товара.

### 3.4.4. Сортировка и вычисление популярных товаров

Система сортировки поддерживает четыре режима:

- `popular`;
- `rating`;
- `price_asc`;
- `price_desc`.

При этом «популярность» рассчитывается не по статическому флагу, а на основании количества заказанных единиц товара. В `catalog.ts` для этого используется SQL-выражение:

```ts
const orderCountExpression = sql<number>`coalesce((
  select sum(${orderItems.quantity})
  from ${orderItems}
  where ${orderItems.productId} = ${products.id}
), 0)`;
```

Схожим образом вычисляются:

- средний рейтинг;
- число отзывов.

После этого на главной странице дополнительно применяется кеширование витрин популярных товаров через `popular_products_cache`. Механизм реализации предполагает периодический пересчёт набора рекомендаций и хранение их результата в БД в течение `POPULAR_PRODUCTS_TTL_MS`.

Такой подход снижает число тяжёлых повторных вычислений и ускоряет выдачу на главной странице.

### 3.4.5. Валидация форм и прикладных данных

Валидация присутствует как на клиенте, так и на сервере, однако решающей является серверная проверка. Это соответствует корректной архитектурной практике, поскольку клиентские ограничения не могут считаться надёжным механизмом защиты.

#### Регистрация и авторизация

При регистрации проверяются:

- непустой логин;
- длина логина от 3 до 50 символов;
- отсутствие пробелов;
- минимальная длина пароля 8 символов;
- длина имени пользователя не более 100 символов;
- уникальность логина.

#### Создание и обновление товара

Серверная логика каталога проверяет:

- допустимость категории;
- положительность цены;
- корректность параметров процессора и видеосистемы;
- ограничение числа изображений;
- обязательность `graphicsModel` для дискретной графики;
- допустимость `null` для неактуальных характеристик.

Пример:

```ts
if (payload.graphicsType === 'discrete' && !payload.graphicsModel) {
  throw new HttpError(400, 'Для дискретной видеокарты требуется указать ее название');
}
```

#### Оформление заказа

При checkout обязательными являются поля:

- ФИО;
- телефон;
- электронная почта;
- адрес доставки.

Проверка выполняется через `requireNonEmptyString(...)`. Если корзина пуста, оформление заказа также блокируется исключением `HttpError(400, 'Корзина пуста')`.

#### Отзывы

Для оценки заказа рейтинг ограничен диапазоном от 1 до 5, а сам отзыв может быть привязан только к товару, действительно входившему в заказ пользователя.

Таким образом, валидация в проекте представляет собой не поверхностную проверку формы, а механизм контроля целостности бизнес-процессов.

### 3.4.6. Логика авторизации и разграничение прав доступа

Проект поддерживает две пользовательские роли:

- `user`;
- `admin`.

Проверка прав реализуется через функции:

- `requireAuthenticatedUser(request)`;
- `requireAdminUser(request)`.

Это позволяет централизовать политику доступа. Например, административные маршруты работы с товарами, заказами и новостями могут быть вызваны только пользователем с ролью `admin`.

Первичная административная учётная запись создаётся автоматически функцией `seedRootAdmin()`. При инициализации базы в таблице `users` гарантируется существование пользователя `root` с ролью администратора. С практической точки зрения это упрощает тестирование административного интерфейса и демонстрацию возможностей системы.

### 3.4.7. Итоговая оценка архитектурного решения

Проведённый анализ показывает, что `TechMarket` реализован как целостное full-stack приложение с ясным разграничением ответственности между уровнями системы. Основные архитектурные достоинства проекта состоят в следующем:

- единая кодовая база для интерфейса, API и доступа к данным;
- типобезопасность на уровне доменных сущностей и HTTP-контрактов;
- реляционная схема данных, отражающая ключевые бизнес-процессы интернет-магазина;
- серверное хранение корзины и сессий;
- транзакционная обработка оформления заказа;
- централизованная валидация и обработка ошибок;
- переиспользуемая компонентная модель интерфейса;
- возможность дальнейшей оптимизации через SSR/ISR и серверные компоненты `Next.js`.

Следовательно, архитектура проекта соответствует задачам разработки современного веб-приложения электронной коммерции и может рассматриваться как практический пример интеграции `Next.js`, `TypeScript`, `Drizzle ORM` и `SQLite` в едином программном решении.
